**4. 数据库设计规范**

**4.1 命名规范**

**必须遵守：** 表名：`<模块名称>_T_<表名称>`；中间表：`<模块名称>_RT_<表名称>`；**主键统一命名为 `id`**；**外键命名为 `xxx_id`**（xxx 为主表逻辑名/简称，如 order_id）；普通索引：`IND_<表名>_N<序号>`；位图索引：`IND_<表名>_B<序号>`；唯一索引：`IND_<表名>_U<序号>`；主键约束：`PK_<表名>`；唯一约束：`UK_<表名>_<字段>`；外键约束：`FK_<表名>_<字段>`；检查约束：`CK_<表名>_<字段>`；使用有意义的英文词汇，避免缩写；不允许使用数据库关键字和保留字

**4.2 建表规范**

**必须遵守：** 不允许将表建立在SYSTEM表空间上；表和索引建立在不同表空间上；建表时必须指明表空间；非空列在前，可空列在后；数据量大且有热数据时可分区；必须设置主键；**必须添加默认字段：** 根据实体基类添加对应的默认字段（详见4.3.0），不得遗漏

**4.3 字段设计规范（基于CCF框架基类）**

**基类字段规范：** 详见技术规则要求第1节实体类设计规则

**字段设计建议：** 主键统一命名为`id`（BIGINT）；时间字段使用`created_time`、`modified_time`（TIMESTAMP WITH TIME ZONE）；用户字段使用`created_user`、`modified_user`（BIGINT）；租户字段使用`tenant_id`（BIGINT）；组织字段使用`org_id`、`org_level_no`、`ent_level_no`；逻辑删除字段使用`is_delete`（BOOLEAN/SMALLINT）；乐观锁字段使用`version`（INT）；字段命名：数据库使用snake_case，Java使用camelCase

**4.3.0 默认字段规则**

**⚠️ 重要：所有继承框架基类的实体表在设计数据库表时，必须在字段表格中完整列出对应的所有默认字段（与业务字段同一表格）。字段顺序：id 放表结构最前，业务字段在中间，其余默认字段放最后。字段名和类型必须与下表完全一致，不得遗漏。**

| createdUser | created_user | ID（通常为BIGINT） | BIGINT | 创建人ID |

- **TenantBaseEntity：** BaseEntity所有字段 + `tenant_id`
- **OrganizationBaseEntity：** TenantBaseEntity所有字段 + `org_id`、`org_level_no`、`ent_level_no`

**字段处理：** 审计字段（created_user、created_time、modified_user、modified_time）由框架自动填充；乐观锁字段（version）由框架自动维护；逻辑删除字段（is_delete）由框架自动处理


**4.3.1 主外键设计规范**

**主键：** 所有主表及子表的主键统一命名为 `id`（BIGINT）；必须设置主键；通常不使用存在实际意义的列做主键；由Sequence产生的ID列不作为组合PK中的列；约束命名格式：`PK_<表名>`

**外键：** 外键字段命名为 `xxx_id`（xxx 为主表逻辑名/简称），如 ord_t_order 主键 id，ord_t_order_item 外键 order_id；用于保证引用完整性，但会增加DML开销；约束命名格式：`FK_<表名>_<字段>`

**唯一性约束：** 允许为空，允许修改；命名格式：`UK_<表名>_<字段>`

**检查约束：** 用于限制特定值（如性别、状态等）；命名格式：`CK_<表名>_<字段>`

**触发子约束：** 一般不使用

**4.4 索引设计规范**

**必须遵守：** 使用普通B树索引；小表（<5000条）不创建索引；单个表索引不超过5个；索引重复率不超过20%；将差别数最多的列放在索引最前面；ORDER BY DESC时创建DESC索引；分区表使用分区索引，分区索引必须包含分区列并指明表空间

**4.5 SQL编写规范**

**查询语句规范：** 表连接不超过8个表；嵌套查询不超过5层；合理使用索引；LIKE通配符%避免出现在字符串开始；NULL值判定会引起全表扫描；使用相同数据类型比较；使用UNION ALL代替UNION；复杂查询使用中间表或临时表优化

**DML语句规范：** 批量操作时先删除索引再重建；大容量操作可暂时禁用约束和触发器；事务尽可能小；不同事务中对表的操作顺序保持一致避免死锁

### 接口定义说明

**标准RESTful接口格式：**

1. **请求方法（HTTP Method）：**
   - `GET`：查询操作
   - `POST`：新建操作
   - `PUT`：修改操作（全量更新）
   - `PATCH`：修改操作（部分更新）
   - `DELETE`：删除操作

2. **请求路径（URL Path）：**
   - 格式：`/api/[模块路径]/[资源路径]`
   - 示例：`/api/dossier/list`、`/api/dossier/{id}`

3. **请求参数类型：**
   - **Query参数：** GET请求的查询参数，通过URL `?key=value` 传递
   - **Path参数：** URL路径中的参数，如 `/api/dossier/{id}`
   - **Body参数：** POST/PUT/PATCH请求的请求体，通常为JSON格式
   - **FormData参数：** 文件上传时使用 `multipart/form-data` 格式

4. **响应格式：**
   - **统一响应结构：**
     ```json
     {
       "code": 200,        // 状态码：200成功，400参数错误，401未授权，403无权限，404不存在，500服务器错误
       "msg": "success",  // 提示信息
       "ok": true,        // 操作是否成功
       "data": {}         // 响应数据，可以是对象{}或数组[]
     }
     ```
   - **分页响应结构：**
     ```json
     {
       "code": 200,
       "msg": "success",
       "ok": true,
       "data": {
         "content": [],           // 数据列表
         "currentPage": 1,         // 当前页码
         "pageSize": 10,           // 每页条数
         "totalElements": 100,     // 总记录数
         "totalPages": 10          // 总页数
       }
     }
     ```

5. **状态码说明：**
   - `200`：操作成功
   - `400`：请求参数错误
   - `401`：未授权，需要登录
   - `403`：无权限访问
   - `404`：资源不存在
   - `413`：请求实体过大（如文件过大）
   - `500`：服务器内部错误

#### 1. 实体类设计规则

**基类继承关系：**
```
IdEntity<ID> (抽象类)
    ↑
BaseEntity<ID> (抽象类)
    ↑
    ├── TreeEntity<ID, T> (树形实体)
    └── TenantBaseEntity<ID> (租户实体)
            ↑
    OrganizationBaseEntity<ID> (组织实体)
```

**基类说明：**

| 基类 | 位置 | 字段 | 主要特性 | 使用场景 |
|------|------|------|----------|----------|
| **IdEntity** | `com.coscoshipping.framework.core.entity.IdEntity` | id（主键ID） | 支持ID脱敏 | 仅需要主键的简单实体 |
| **BaseEntity** | `com.coscoshipping.framework.core.entity.BaseEntity` | id、createdUser、createdTime、modifiedUser、modifiedTime、version、delete | 审计、乐观锁、逻辑删除 | 标准业务实体（推荐默认选择） |
| **TreeEntity** | `com.coscoshipping.framework.core.entity.TreeEntity` | 继承BaseEntity + ITree接口方法 | 树形结构、排序、自动构建 | 组织架构、菜单、分类等树形数据 |
| **TenantBaseEntity** | `com.coscoshipping.framework.mybatisflex.entity.TenantBaseEntity` | 继承BaseEntity + tenantId | 租户隔离、自动注入 | SaaS多租户应用 |
| **OrganizationBaseEntity** | `com.coscoshipping.framework.mybatisflex.entity.OrganizationBaseEntity` | 继承TenantBaseEntity + orgId、orgLevelNo、entLevelNo | 组织关联、层级管理、数据权限 | 需要组织权限控制的业务实体 |

**基类选择指南：**

| 场景 | 推荐基类 | 说明 |
|------|----------|------|
| 仅需要主键的简单实体 | IdEntity | 最基础的实体类 |
| 标准业务实体（单体应用） | BaseEntity | 推荐作为默认选择 |
| 需要树形结构的实体 | TreeEntity | 菜单、组织等树形数据 |
| 多租户SaaS应用 | TenantBaseEntity | 需要租户隔离 |
| 需要组织权限控制 | OrganizationBaseEntity | 需要组织数据权限 |
| 树形+租户 | TenantBaseEntity + ITree | 树形数据且需要租户隔离 |
| 树形+组织 | OrganizationBaseEntity + ITree | 树形数据且需要组织权限 |

**必须遵守：**
- 所有业务实体必须继承框架提供的基类（IdEntity、BaseEntity、TreeEntity、TenantBaseEntity、OrganizationBaseEntity）
- 使用MyBatis-Flex时，实体必须继承对应的BaseModel基类（BaseModel、IdModel、TenantBaseModel、OrganizationBaseModel）
- 实体必须使用 `@Table("table_name")` 注解指定表名
- 字段必须使用 `@Column("column_name")` 注解指定列名
- 主键生成使用 `@TableId` 注解配置（默认雪花算法）
- 审计字段（createdUser、createdTime、modifiedUser、modifiedTime）由框架自动填充，不应手动设置
- 乐观锁字段（version）由框架自动维护，更新数据时需要正确处理
- 逻辑删除字段（delete）由框架自动处理，查询时自动过滤已删除数据
- 树形结构实体必须继承 `TreeEntity<ID, T>` 并实现 `ITree<T>` 接口
- 记录状态使用 `RecordState` 枚举，不使用自定义状态值
- 主键字段统一命名为 `id`
- 时间字段使用 `created_time`、`modified_time`（数据库字段名）
- 用户字段使用 `created_user`、`modified_user`（数据库字段名）
- 租户字段使用 `tenant_id`（数据库字段名）
- 组织字段使用 `org_id`、`org_level_no`、`ent_level_no`（数据库字段名）
- 逻辑删除字段使用 `is_delete`（数据库字段名，建议类型为tinyint或smallint）

**禁止行为：**
- 禁止直接使用 `new PageRequest()`，必须使用 `PageRequest.of()`
- 禁止在实体类中直接定义id字段，必须继承IdEntity
- 禁止使用JPA注解（如 `@Entity`、`@Table`），必须使用MyBatis-Flex注解
- 禁止在实体类中定义业务逻辑方法
- 禁止手动设置审计字段（自动填充）
- 禁止手动设置乐观锁版本号（由框架自动维护）
- 禁止手动设置逻辑删除标记（由框架自动处理）
- 禁止在实体类中使用自定义的审计字段名称
- 禁止在实体类中使用自定义的逻辑删除字段名称

#### 2. Controller层开发规则

**必须遵守：** 使用`@RestController`注解；返回`HttpResult<T>`类型；继承框架提供的CRUD控制器基类；参数验证使用`@Valid`或`@Validated`；ID参数脱敏使用`@IdMask`；异常抛出由全局异常处理器处理

**禁止行为：** 禁止直接返回实体对象；禁止在Controller中处理异常；禁止直接操作数据库；禁止处理业务逻辑；禁止使用自定义响应格式


#### 3. Service层开发规则

**必须遵守：** 继承`BaseService<T>`或实现`IBaseService<T>`；使用`@Service`注解；使用框架提供的工具方法（queryColumn、fieldName、PageUtils等）；事务处理使用`@Transactional`注解

**禁止行为：** 禁止直接使用Mapper；禁止直接使用SQL语句；禁止忽略异常处理


#### 4. 查询参数规则

**必须遵守：** 分页查询使用`PageQueryParam<T>`；列表查询使用`ListQueryParam<T>`；存在性检查使用`ExistParam`；范围查询使用`RangeValue`；排序使用`SortEnum`枚举；排序字段通过`@AllowOrderByFields`控制；分页使用`PageRequest.of(page, size)`

**禁止行为：** 禁止使用自定义分页参数类；禁止在查询参数中使用SQL语句；禁止使用魔法值作为排序方向

#### 5. 参数验证规则

**必须遵守：** Controller参数使用`@Valid`或`@Validated`；实体字段使用验证注解（@NotNull、@NotBlank、@Mobile、@Xss等）；列表参数使用`ValidList<T>`；参数验证使用`Assert`工具类

**禁止行为：** 禁止手动验证参数；禁止使用`if (obj == null)`，必须使用`Assert.notNull()`；禁止跳过参数验证


#### 6. 异常处理规则

**必须遵守：** 业务异常继承`BaseException`或使用`ServiceException`；异常包含错误码和错误消息；使用`FrameworkExceptions`枚举；异常抛出由全局异常处理器处理

**禁止行为：** 禁止直接抛出`RuntimeException`；禁止在Controller中处理异常；禁止暴露敏感信息；禁止使用空的异常消息


#### 7. 数据权限规则

**必须遵守：** 实现`DataPermissionScopeProvider`接口；数据权限应用到所有数据查询；多租户实体继承`TenantBaseModel<T>`；多组织实体继承`OrganizationBaseModel<T>`

**禁止行为：** 禁止忽略数据权限；禁止忽略租户隔离；禁止手动设置租户ID和组织ID；禁止硬编码组织范围

#### 8. 工具类和常量使用规则

**必须遵守：** 字符串常量使用`StringConstants`；字符常量使用`CharConstants`；SQL关键字使用`SqlConstants`；HTTP方法使用`HttpMethod`枚举；日期格式化使用`DateFormatUtils`或`OffsetDateTimeUtils`；树形转换使用`TreeUtils.buildTree()`；上下文使用`ContextHolder`；跟踪ID使用`TraceIdHelper.getTraceId()`

**禁止行为：** 禁止硬编码字符串常量；禁止直接使用字符串表示HTTP方法；禁止手动实现树形转换；禁止使用ThreadLocal直接存储上下文；禁止手动生成跟踪ID

#### 9. 数据库设计规则

**必须遵守：** 表名格式`<模块名称>_T_<表名称>`；中间表格式`<模块名称>_RT_<表名称>`；索引命名`IND_<表名>_N<序号>`等；约束命名`PK_<表名>`、`UK_<表名>_<字段>`等；必须设置主键；建表时指明表空间；非空列在前，可空列在后；单个表索引不超过5个；小表（<5000条）不创建索引；表和字段必须添加注释；SQL查询表连接不超过8个表；嵌套查询不超过5层；**⚠️ 建表时必须添加默认字段：** 根据实体类继承的基类，必须在数据库表中添加对应的默认字段（详见4.3.0默认字段规则），字段名和类型必须完全一致，不得遗漏；基类字段使用框架规定的字段名和数据类型；主键统一命名为`id`（BIGINT类型）；审计字段使用`created_user`、`created_time`、`modified_user`、`modified_time`；乐观锁字段使用`version`（INT类型）；逻辑删除字段使用`is_delete`（BOOLEAN/SMALLINT类型）；多租户系统必须为`tenant_id`创建索引；组织权限系统必须为`org_id`创建索引

**禁止行为：** 禁止使用数据库关键字和保留字；禁止将表建立在SYSTEM表空间；禁止创建超过5个索引；禁止在SQL查询中连接超过8个表；禁止嵌套查询超过5层；禁止在LIKE运算符字符串开始使用通配符%；禁止使用不同类型数据比较；禁止不添加注释
